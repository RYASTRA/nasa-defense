# Security Policy

## Supported Versions

`nasa-defense` is a continuously-deployed application, not a released library. There is
one running instance — the scheduled GitHub Actions workflow on `main`. Fixes land on
`main`; there are no maintained release branches or version tags.

| Ref                        | Supported          |
| -------------------------- | ------------------ |
| `main` (deployed)          | :white_check_mark: |
| forks / older commits      | :x:                |

## Reporting a Vulnerability

Please report security issues **privately** — do not open a public Issue. Issues in this
repo are auto-generated by the watcher and world-readable, so a report filed there is
immediately public.

Use GitHub's [private vulnerability reporting][report] ("Report a vulnerability" under the
**Security** tab). Expect an initial acknowledgement within a week; this is a personal
project maintained on a best-effort basis, so please allow time for a fix before any
public disclosure.

[report]: https://github.com/RYASTRA/nasa-defense/security/advisories/new

## Security model

What is and isn't security-relevant in this project:

- **Secrets.** The watcher reads `NASA_API_KEY`, `GITHUB_TOKEN`, and optional fan-out
  credentials (`SLACK_WEBHOOK_URL`, `DISCORD_WEBHOOK_URL`, `SMTP_*`) from GitHub Actions
  secrets, or from a local `.env` that is gitignored and never committed. No secret is
  written to committed state, run logs, or the published site.
- **Token scope.** CI runs with the default `GITHUB_TOKEN`, scoped to this repository
  only. It writes Issues, pushes state commits to `main`, and deploys the Pages site. It
  has no organization-wide or cross-repository access.
- **Outbound traffic.** The app makes outbound HTTPS requests only to NASA/JPL APIs
  (`api.nasa.gov`, `ssd-api.jpl.nasa.gov`) and to any fan-out webhooks you configure. It
  accepts no inbound network input.
- **Dependencies.** The only runtime dependency is `httpx`; updates are tracked by
  Dependabot.
